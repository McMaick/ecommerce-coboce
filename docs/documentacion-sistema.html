Descripción general del sistema
1.1 Objetivo y alcance

El sistema es una plataforma de comercio electrónico desarrollada para la Distribuidora Cerámica COBOCE, ubicada en Cobija, Bolivia. Permite a los clientes explorar el catálogo de productos, realizar compras en línea, acumular y canjear puntos de fidelidad, y recibir sus pedidos mediante un servicio de delivery propio.

🛒

Tienda en línea

Catálogo con búsqueda, filtros por categoría, detalle de producto, lista de deseos y carrito.

⭐

Programa de puntos

Los clientes ganan puntos por cada compra y los canjean como descuento en futuros pedidos.

🚚

Gestión de delivery

Asignación de repartidores, seguimiento de estado y notificación WhatsApp automática.

Roles de usuario

El sistema maneja tres roles diferenciados, controlados por el campo rol_id de la tabla usuarios:

rol_idRolAccesoPermisos principales
1 Administrador /admin/* CRUD completo de productos, categorías, pedidos, usuarios; configurar puntos; ver reportes; asignar repartidores

3.1 Autenticación

Gestionado por controllers/AuthController.php. Incluye registro, inicio de sesión, cierre de sesión y protección de páginas por rol.

RegistroCrea una cuenta nueva, asigna rol_id = 2 (cliente), otorga 50 puntos de bienvenida automáticamente.

3.3 Carrito de compras

Manejado por la tabla carrito en la base de datos, vinculado al usuario autenticado. Funcionalidades principales:

  • Agregar productos con cantidad seleccionada
  • Actualizar cantidades o eliminar ítems
  • Verificación de stock en tiempo real al agregar
  • El badge del carrito en el header muestra el total de ítems
  • Los ítems persisten entre sesiones (guardados en BD)

PasoArchivoQué ocurre
1 — Entrega paso1-entrega.php El cliente elige delivery (con zona y dirección) o retiro en tienda. Las zonas se cargan desde zonas_delivery con su costo. Se guarda en $_SESSION['checkout']. Se limpia cualquier confirmación anterior.
2 — Puntos paso2-puntos.php Si el cliente tiene puntos, puede elegir cuántos canjear. El sistema calcula automáticamente el descuento máximo (30% del total). El saldo y la decisión quedan en sesión.
3 — Pago paso3-pago.php El cliente selecciona método de pago (efectivo, QR, transferencia). Si paga con transferencia/QR puede subir el comprobante. El código QR se genera dinámicamente.
4 — Confirmación paso4-confirmacion.php Se crea el pedido en BD dentro de una transacción: se descuenta stock, se registran movimientos de puntos (ganados y canjeados), se genera el código único del pedido (ej. COB9338-20260502), se vacía el carrito.
Código de pedido: Se genera con formato COB{4_chars_random}-{AAAAMMDD}. Cada pedido tiene un código único almacenado con restricción UNIQUE en la BD.
3.5 Sistema de puntos de fidelidad

Configurado desde el panel admin en admin/puntos.php. Los valores se guardan en la tabla config_puntos y son editables sin tocar código.

ParámetroValor por defectoDescripción
Puntos por Bs. gastado1.00 pt / Bs.Por cada boliviano del subtotal, el cliente gana 1 punto.
Valor de cada puntoBs. 0.10Al canjear, 1 punto = Bs. 0.10 de descuento.
Descuento máximo30%El canje no puede superar el 30% del total del pedido.
Bonus de bienvenida50 puntosSe otorgan automáticamente al registrarse.

Tipos de movimientos de puntos:

  • ganado — Compra completada y entregada
  • canjeado — Uso de puntos como descuento en checkout
  • ajuste — Corrección manual por admin
  • vencimiento — Expiración de puntos (reservado)
Reversión al cancelar: Si un pedido es cancelado, el sistema devuelve automáticamente los puntos canjeados al cliente y revierte los puntos que hubiera ganado, manteniendo la integridad del saldo.

Sincronización de estados: Cuando el admin cambia el estado de un pedido en admin/pedidos.php, el sistema actualiza automáticamente también el estado en asignaciones_delivery para mantener consistencia en el historial del repartidor.

Seguridad
MecanismoImplementación
Contraseñas Hasheadas con password_hash() (bcrypt, costo 12). Verificación con password_verify(). Nunca se almacena la contraseña en texto plano.
CSRF Token único por sesión generado con random_bytes(32), incluido en todos los formularios POST y verificado antes de procesar.
SQL Injection Todas las consultas usan PDO con parámetros preparados (prepare() + execute()). Ninguna consulta concatena variables directamente.
XSS Toda salida de datos de usuario pasa por limpiar() que aplica htmlspecialchars() con flags ENT_QUOTES.
Sesiones Cookie httponly, SameSite=Lax, lifetime 24h. Se llama session_regenerate_id(true) al autenticarse para prevenir session fixation.
Control de acceso Cada página protegida verifica $_SESSION['usuario_rol']. El admin panel requiere rol_id = 1 explícitamente.
Subida de archivos Validación de extensión (JPG, PNG, WEBP), tamaño máximo 3MB, nombre aleatorio con uniqid() para evitar sobrescrituras.
Transacciones BD Las operaciones críticas (crear pedido, actualizar stock, registrar puntos) se ejecutan dentro de transacciones PDO con rollback en caso de error.
7
Reglas de negocio

Stock

El stock se descuenta al confirmarse el pedido (paso 4), no al agregarlo al carrito. Si el stock es insuficiente al confirmar, la transacción falla.

Puntos — ganancia

Los puntos se acreditan al cliente cuando el admin marca el pedido como Entregado, no al confirmar la compra.

Puntos — cancelación

Al cancelar un pedido, el sistema devuelve puntos canjeados y revierte puntos ganados automáticamente.

Checkout continuo

Al iniciar un nuevo checkout (paso 1), se limpia la sesión del pedido anterior para evitar que un cliente vea la confirmación de una compra antigua.

Soft delete

Los productos no se eliminan físicamente. Se marcan con activo = 0 para preservar el historial de pedidos. Pueden restaurarse.

Delivery sincronizado

Al cambiar el estado de un pedido en el admin, el estado en asignaciones_delivery se actualiza en la misma transacción.

Flujos principales
8.1 Flujo de compra completo
  1. Cliente navega el catálogo y agrega productos al carrito
  2. Hace clic en "Ir al checkout" → se verifica que esté autenticado
  3. Paso 1: Elige tipo de entrega, zona y dirección
  4. Paso 2: Decide si canjea puntos (slider de cantidad)
  5. Paso 3: Selecciona método de pago, sube comprobante si aplica
  6. Paso 4: Se ejecuta la transacción: crea pedido, descuenta stock, registra movimientos de puntos, limpia carrito
  7. Cliente ve resumen de confirmación con código de pedido
  8. Admin recibe el pedido en su panel y lo procesa
8.2 Flujo de asignación de delivery
  1. Admin entra al detalle del pedido en admin/pedidos.php
  2. En el panel lateral "Repartidor", selecciona al repartidor disponible y agrega notas opcionales
  3. Clic en "Asignar repartidor"
  4. El sistema crea el registro en asignaciones_delivery, avanza el estado del pedido a En camino
  5. Se genera el mensaje WhatsApp con todos los detalles del pedido
  6. Aparece un modal con la vista previa del mensaje y el botón "Abrir WhatsApp y enviar"
  7. Admin hace clic → WhatsApp se abre con el mensaje pre-llenado para el repartidor
  8. Repartidor recibe el mensaje, recoge el pedido y lo entrega
  9. Admin confirma la entrega cambiando el estado a "Entregado" → se acreditan los puntos al cliente
8.3 Flujo del sistema de puntos
  1. Cliente se registra → recibe 50 puntos de bienvenida
  2. Cliente hace una compra de Bs. 200 → el sistema calcula 200 puntos a ganar (si config es 1 pt/Bs.)
  3. En checkout Paso 2: si tiene puntos, puede canjear hasta el 30% del total
  4. Al confirmar el pedido: se descuentan los puntos canjeados del saldo
  5. Al marcar como entregado: se acreditan los puntos ganados del pedido

Configuración e instalación
9.1 Requisitos
  • XAMPP (Apache 2.4 + PHP 8.x + MySQL 8.0+)
  • Navegador moderno (Chrome, Firefox, Edge)
  • Acceso a internet para cargar Bootstrap y Google Fonts vía CDN
9.2 Pasos de instalación
  1. Copiar la carpeta ecommerce-coboce/ en C:\xampp\htdocs\
  2. Iniciar Apache y MySQL desde el panel de XAMPP
  3. Abrir phpMyAdmin en localhost/phpmyadmin
  4. Crear la base de datos ecommerce_coboce
  5. Importar el archivo database/coboce.sql
  6. Verificar config/config.php: las credenciales de BD son root sin contraseña (XAMPP por defecto)
  7. Acceder a localhost/ecommerce-coboce
